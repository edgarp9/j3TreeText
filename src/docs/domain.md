# j3TreeText 도메인 요구사항

## 목표

j3TreeText는 SQLite 파일에 저장된 트리형 문서를 편집하는 데스크톱 앱과 CLI다. GUI와 CLI는 Windows와 Linux 빌드를 지원할 수 있도록 OS 전용 처리를 `platform` 경계 아래에 둔다. 현재 구현 범위는 앱 시작, DB 생성/마이그레이션, 트리 표시, 문서 탭 편집/저장, 노드 CRUD, 휴지통, 검색, 트리 이동/정렬, 드래그 앤 드롭, 저장 충돌 방지, UI 상태 복원을 포함한다.

## 구현 책임 경계

- `main.rs`: 앱 시작과 최상위 오류 표시를 담당한다. Windows 릴리즈 빌드의 GUI 실행 파일은 콘솔 창을 만들지 않는 GUI 서브시스템으로 빌드한다.
- `app`: 유스케이스 흐름과 repository 호출 순서를 조합한다.
- `domain`: 공통 도메인 오류와 순수 모델을 담당한다. 내부 구현은 `document_tree`, `ui_settings`, `text_edit`, `open_tabs` 모듈로 나누어 문서 트리 규칙, UI 설정 저장값 파싱, 텍스트 찾기/바꾸기, 열린 문서 탭 세션 상태의 변경 범위를 분리한다.
- `infra/sqlite`: SQLite schema, migration, repository 트랜잭션과 쿼리를 담당한다.
- `infra/desktop_entry`: Linux 사용자 영역 desktop entry와 hicolor 아이콘 설치/제거를 담당한다. 앱 시작 시 자동 등록하지 않고, GUI 실행 파일의 명시적 `--install`/`--uninstall` 명령에서만 파일 I/O를 수행한다.
- `platform/gui`: OS별 GUI 백엔드를 선택하는 경계다. Windows는 `platform/win32` 백엔드를 사용하고, Linux는 GTK4 백엔드를 사용한다. 두 GUI는 같은 `app`, `domain`, `infra` 유스케이스를 호출한다. 메뉴와 우클릭 메뉴의 명령 목록, 상황별 활성화 정책, 키보드 단축키 scope는 `platform/gui/command_contract.rs`의 공통 계약을 기준으로 Win32 command id와 GTK action에 매핑한다.
- `platform/file_system`: OS별 파일 교체 방식을 담당한다. Windows는 Win32 `MoveFileExW` FFI를 사용하고, Linux/Unix는 `std::fs::rename` 기반 원자 교체를 사용한다.
- `platform/win32`: `windows-sys` 기반 Windows UI와 `unsafe` FFI 경계를 담당한다. 내부는 창/메시지 루프(`window`), 상태 모델(`state`), UTF-16 텍스트 경계(`text`), 배치와 UI 설정(`layout`), TreeView 처리(`tree`), 메뉴 명령과 저장/dirty 흐름(`commands`)으로 나눈다.
- `platform/gui/linux`: GTK4 기반 Linux UI를 담당한다. 메뉴와 액션은 `gio::SimpleAction`, 왼쪽 문서 영역은 GTK 위젯 계층, 본문 편집은 `TextView`/`TextBuffer`, 파일/글꼴 선택과 확인 메시지는 GTK 대화상자 경계를 사용한다. 플랫폼별 위젯 타입은 이 모듈 안에 고립하고 문서 본문과 탭 상태는 공통 모델을 사용한다.

사용자에게 표시되는 오류 메시지는 `AppError::user_message()`에서 분리해 제공하고, 내부 원인은 `Display`와 error source 체인에 남긴다. 플랫폼 오류는 앱 시작 실패와 실행 중 UI 작업 실패를 구분해 안내한다. 공통 `AppError` fallback은 특정 OS 이름을 섞지 않는 데스크톱 UI 메시지로 유지하고, Win32/Linux GUI 내부 대화상자는 각 플랫폼 i18n 경계에서 더 구체적인 문구를 제공한다.

## 용어

- 문서 저장소: 실행 파일과 같은 폴더에 놓이는 SQLite DB 파일.
- Linux application id: GTK application id, desktop entry id, hicolor 아이콘 이름에 사용하는 `io.github.edgarp9.j3TreeText` 값이다.
- 노드: 트리에 표시되는 기본 항목이며 항상 문서다. 모든 노드는 본문과 하위 문서를 함께 가질 수 있다.
- 문서 노드: 사용자가 편집할 본문을 가지며, 다른 문서의 부모가 될 수 있는 항목이다.
- 열린 문서 탭: 오른쪽 편집 영역에 열린 문서 노드의 편집 세션이다. 각 탭은 `node_id`, 제목, 현재 본문, 로드 당시 `updated_at`, dirty 상태, 편집 가능 여부, 편집기 표시 위치를 독립적으로 가진다.
- dirty 탭: 현재 본문이 마지막 로드 또는 저장된 본문과 다른 열린 문서 탭이다.
- 루트 문서: 최초 실행 시 생성되는 최상위 문서다. 트리에 표시되고 편집 가능하지만 삭제하거나 이동할 수 없다.
- 휴지통: `deleted_at IS NOT NULL`인 삭제 노드를 확인하고 복원 또는 영구 삭제하는 보기다.

## 노드 규칙

- `title`은 공백만 있는 값을 허용하지 않는다.
- `sort_order`는 같은 부모 아래에서 표시 순서를 정하는 0 이상의 정수다.
- 루트 노드는 부모가 없고 문서 본문을 가진다.
- 모든 노드는 `content`를 가지며, 새 문서는 빈 문자열을 저장한다.
- `deleted_at IS NULL`인 활성 노드 기준으로 같은 부모 아래 동일한 `title`은 허용하지 않는다.
- 제목 중복 비교는 현재 SQLite 기본 `TEXT` 비교를 따른다. 즉 대소문자를 구분한다.
- `새 문서`는 선택된 문서의 형제로 만든다. 선택된 문서가 루트이면 루트 아래에 만든다.
- `새 하위 문서`는 선택된 문서 아래에 만든다.
- 새 문서 기본 이름은 `Untitled`이며, 기본 이름이 이미 있으면 `Untitled 2`처럼 중복되지 않는 제목을 사용한다.
- 이름 변경 시 입력 제목은 앞뒤 공백을 제거한 뒤 저장한다.
- 트리뷰 라벨 편집 중 포커스를 잃으면 현재 입력 텍스트로 이름 변경을 확정한다. `Esc`처럼 사용자가 명시적으로 취소한 편집은 저장하지 않는다.

## 최초 실행 정책

DB 파일이 없거나 `nodes` 테이블이 비어 있으면 앱 시작 시 다음 초기 노드를 생성한다.

- `id = 1`, `title = Root`, 부모 없음, 빈 본문
- `id = 2`, `title = Untitled`, 부모는 루트 문서, 빈 본문

이미 노드가 하나 이상 있으면 자동 초기 노드를 다시 만들지 않는다.

## Linux desktop entry 설치 정책

Linux GUI 실행 파일은 다음 CLI 형태를 지원한다.

- 인자 없음: 기본 문서 저장소로 기존 GUI를 실행한다.
- 설정 파일 인자 1개: 지정한 DB 파일을 문서 저장소로 사용해 GUI를 실행한다.
- `--install`: 사용자 영역 desktop entry와 아이콘을 설치하고 종료한다.
- `--uninstall`: `--install`이 설치한 desktop entry와 아이콘을 제거하고 종료한다.

`--install`은 `$XDG_DATA_HOME`이 절대 경로이면 그 값을 사용하고, 없으면 `$HOME/.local/share`를 사용한다. 주 desktop entry는 `$DATA_HOME/applications/io.github.edgarp9.j3TreeText.desktop`에 쓰고, 아이콘은 SVG가 있으면 `$DATA_HOME/icons/hicolor/scalable/apps/io.github.edgarp9.j3TreeText.svg`를 우선 설치한다. SVG가 없고 `icon.png`가 있으면 `$DATA_HOME/icons/hicolor/256x256/apps/io.github.edgarp9.j3TreeText.png`를 fallback으로 설치한다. 원본 아이콘은 실행 파일 폴더, 현재 작업 폴더 순서로 찾는다.

Linux application id에 ASCII 대문자가 있으므로 KDE/Plasma taskbar fallback을 위해 lowercase alias인 `io.github.edgarp9.j3treetext` desktop entry와 아이콘도 함께 설치한다. Alias desktop entry에는 앱 메뉴 중복을 막기 위해 `NoDisplay=true`를 넣는다. `--uninstall`은 주 id, lowercase alias, legacy id(`io.github.j3treetext`, `j3TreeText`, `j3treetext`)의 `.desktop`, `.svg`, `.png` 파일을 모두 제거하며, 이미 없는 파일은 성공으로 처리한다.

GTK application id와 기본 창 아이콘 이름은 `io.github.edgarp9.j3TreeText`로 유지한다. 개발 실행이나 직접 실행에서도 창 아이콘이 보이도록 GTK 시작 시 실행 파일 폴더와 현재 작업 폴더에서 `icon.svg`, `icon.png` 순서로 런타임 아이콘 파일을 찾고, 찾은 파일을 창 realized 시점의 GDK toplevel icon list에도 설정한다.

## 라이선스 고지 정책

프로젝트 라이선스는 `GPL-3.0-or-later`이며, 배포 산출물에는 프로젝트 `LICENSE`를 포함한다. 바이너리를 공개 배포할 때는 해당 빌드의 대응 소스를 GPL-3.0-or-later 조건으로 제공해야 한다.

외부 Rust 의존성 고지는 잠긴 `Cargo.lock` 그래프를 기준으로 관리한다. `tools/generate_third_party_notices.py`는 `cargo metadata --locked --format-version 1`에서 workspace 외부 패키지를 수집하고, 요약형 `THIRD_PARTY_NOTICES.txt`를 생성한다. 고지 파일은 대응 소스 안내, 라이선스 표현식 요약, `Third-Party Component Summary`, 배포물에 포함되지 않는 CI/빌드 도구, Cargo 패키지 inventory, 라이선스 계열별 목록, 대표 라이선스 전문을 포함한다. Cargo 밖에 포함된 Google Material Symbols/Icons 리소스는 upstream Apache-2.0 고지와 함께 별도 항목으로 명시한다.

릴리스 바이너리에는 `LICENSE`, `about.txt`, `THIRD_PARTY_NOTICES.txt`를 함께 포함한다. `build_release.py`는 릴리스 빌드, 고지 파일 복사, 소스/바이너리 zip 검증을 완료한 뒤 기본적으로 `target/release` 폴더를 열며, `--no-open` 인자가 있으면 열지 않는다. About 대화상자는 실행 파일 옆 `about.txt`를 우선 표시하고, 파일을 읽을 수 없으면 빌드에 포함된 동일한 기본 내용을 표시해 프로젝트 라이선스, 라이선스 전문 위치, 소스코드 제공 안내, 제3자 고지 파일 위치, GPL 대응 소스 제공 의무를 안내한다.

`rusqlite`는 `libsqlite3-sys`의 `bundled` feature를 사용하므로 릴리스 빌드에 SQLite 소스가 포함될 수 있다. SQLite의 public-domain 상태와 배포 채널별 증빙 필요성은 릴리스 전 법무/배포 정책에서 확인한다. Linux GUI 빌드는 `pkg-config`가 찾는 GTK 네이티브 라이브러리에 링크한다. OS 패키지에 의존하지 않고 GTK stack을 함께 번들링하는 배포 방식이면 해당 네이티브 라이브러리의 라이선스 원문과 재배포 의무를 별도로 포함한다.

## 삭제 정책

삭제는 물리 삭제가 아니라 `deleted_at`에 삭제 시각을 기록하는 소프트 삭제를 기본 정책으로 둔다. 조회 시 삭제된 노드는 제외한다.

문서를 삭제하면 해당 문서와 모든 하위 활성 문서를 같은 삭제 시각으로 함께 soft delete 처리한다. 루트 문서는 삭제할 수 없다.

삭제 트랜잭션은 삭제 대상의 원래 부모 아래에 남은 활성 자식의 `sort_order`를 0부터 다시 계산해 저장한다.

삭제, 복원, 영구 삭제, 덮어쓰기처럼 Yes/No 확인만 필요한 GUI 대화상자는 Windows `MessageBoxW`의 `MB_YESNO` 기준을 따른다. 버튼 의미와 표시 순서는 Yes/No이며, 키보드 기본 응답은 Yes이다. Linux GTK 대화상자도 이 순서와 기본 응답을 유지한다.

## 휴지통, 복원, 영구 삭제 정책

휴지통 보기는 삭제된 노드만 표시한다. 삭제된 노드의 원래 부모가 함께 삭제되어 있으면 휴지통 안에서는 원래 계층으로 표시한다. 원래 부모가 활성 상태이거나 존재하지 않아 휴지통 안에 함께 표시되지 않는 삭제 노드는 휴지통 최상위에 표시한다.

휴지통의 문서 본문은 확인할 수 있지만 편집할 수 없다. 새 문서/새 하위 문서/이름 변경/이동은 활성 트리에서만 가능하다.

복원은 repository 트랜잭션으로 처리한다. 선택한 삭제 노드와 삭제 상태인 하위 노드를 함께 복원한다. 원래 부모 문서가 활성 상태이면 원래 부모 아래로 복원하고, 원래 부모가 없거나 삭제 상태이면 루트 문서 아래로 복원한다.

복원 대상 부모 아래에 같은 `title`의 활성 노드가 이미 있으면 복원되는 최상위 노드 제목에 ` (restored)`를 붙인다. 그 제목도 충돌하면 ` (restored) 2`, ` (restored) 3`처럼 숫자를 붙여 활성 형제 제목과 충돌하지 않게 한다.

영구 삭제는 repository 트랜잭션으로 처리한다. 휴지통에서 선택한 노드와 모든 하위 노드를 DB에서 물리 삭제한다. 영구 삭제된 노드는 복원할 수 없다.

## 이동 및 정렬 정책

활성 노드만 이동할 수 있다. 삭제된 노드는 이동 대상과 드롭 대상에서 제외한다.

루트 문서는 이동할 수 없다. 모든 문서는 자기 자신이나 자기 하위 노드 아래로 이동할 수 없다. 이 검사는 repository 트랜잭션 안에서 현재 활성 트리를 기준으로 수행한다.

트리뷰 드래그 앤 드롭 정책은 다음과 같다.

- 문서 위에 드롭하면 드래그한 노드를 해당 문서의 마지막 자식으로 이동한다.
- 빈 영역이나 자기 자신 위에 드롭하면 이동하지 않는다.
- 노드 이동은 앱 내부 트리뷰에서 시작한 드래그에만 반응한다. 외부 앱이나 다른 위젯에서 들어온 텍스트/파일 drop은 노드 id처럼 보이더라도 문서 이동으로 처리하지 않는다.
- 검색 결과 보기와 휴지통 보기에서는 드래그 이동을 허용하지 않는다.
- 이동 전 dirty 탭이 있으면 기존 저장 확인 흐름을 먼저 수행한다.
- 이동 후 열린 탭의 dirty 내용은 보존하고, 이동된 문서가 선택 상태로 유지되도록 트리를 새로 고친다. 드래그 앤 드롭으로 부모가 바뀐 열린 dirty 탭도 active document metadata 기준으로 부모와 저장 충돌 기준 토큰을 갱신한다.

같은 부모 아래 정렬 순서는 `sort_order ASC, title ASC, id ASC` 기준으로 조회하고 표시한다. 이동이나 정렬 보조 기능이 실행되면 해당 부모의 활성 자식 `sort_order`를 0부터 다시 계산해 저장한다.

트리뷰에 포커스가 있을 때 `Ctrl+Up`과 `Ctrl+Down`은 선택된 노드를 같은 부모 아래에서 한 칸 위 또는 아래로 이동한다. 같은 동작은 `문서 > 위로`, `문서 > 아래로` 및 트리 우클릭 메뉴에서도 실행할 수 있다. 이동 후 열린 탭의 local content는 유지하되 active document metadata 기준으로 제목, 부모, 저장 충돌 기준 토큰을 갱신한다.

휴지통 이동은 선택한 노드와 하위 노드의 열린 탭을 닫고, 남은 열린 탭은 active document metadata 기준으로 다시 동기화한다. 삭제로 sibling order가 바뀐 문서가 dirty 열린 탭이면 본문은 보존하면서 저장 충돌 기준 토큰을 최신 metadata로 갱신한다.

트리뷰에 포커스가 있을 때 `Delete`는 활성 트리에서는 휴지통 이동, 휴지통 보기에서는 영구 삭제를 실행한다. 확인 대화상자를 취소하거나 오류가 발생해도 Delete 단축키 처리가 끝나면 트리뷰 포커스를 복원해 Windows와 같은 키보드 탐색 흐름을 유지한다.

## 저장 정책

UI 설정은 앱 종료, 관련 설정 변경, 창 크기 변경, splitter 조작 경계에서 저장한다. Windows는 이동/크기 조절 루프나 splitter drag가 끝날 때 현재 창 크기, splitter, 마지막 활성 트리 선택을 함께 저장한다. Linux GTK4는 GDK surface 크기 변경과 splitter 위치 변경을 짧게 debounce한 뒤 같은 설정 묶음을 저장하고, 앱 종료 시 실제 창 크기로 한 번 더 확정 저장한다. 창 크기는 현재 창에서 유효한 양수 크기를 읽을 수 있을 때만 갱신하고, 0 이하처럼 유효하지 않은 크기에서는 기존 값을 보존한다. 저장된 창 위치는 Windows에서 창 배치 복원에 사용한다. Linux GTK4는 창 위치 조회와 강제 이동 API를 제공하지 않으므로 위치 값은 보존하되 크기와 splitter, 선택 상태를 저장·복원한다.

노드는 SQLite `nodes` 테이블에 저장한다. 생성 시각과 수정 시각은 ISO-8601 UTC 문자열로 저장한다. 문서 저장 시 `content`와 `updated_at`을 함께 갱신한다.

문서 노드의 본문은 plain text이며, UTF-8 문자열로 repository 계층을 통해 SQLite `nodes.content`에 저장한다. Windows UI 경계에서는 본문 편집기 전용 Rich Edit 컨트롤의 UTF-16 텍스트를 UTF-8 plain text로 변환한 뒤 저장하고, DB에서 읽은 UTF-8 본문은 UTF-16 plain text로 변환해 편집기에 표시한다. Linux GTK4 경계에서는 `TextBuffer`의 UTF-8 텍스트를 앱 내부 `String`과 동기화한다. Rich Edit 컨트롤이나 GTK `TextView`를 사용하더라도 RTF나 글자별 서식은 문서 본문 모델에 저장하지 않는다. Windows 본문 Rich Edit 컨트롤은 생성 직후 plain text 모드로 고정하고, 기존 `CONTROL_EDITOR_ID`와 `EN_CHANGE` dirty 추적 경로가 유지되도록 Rich Edit 변경 알림 마스크와 표준 텍스트 길이 제한을 함께 설정한다. 검색창과 트리 라벨 편집은 플랫폼별 기본 텍스트 입력 위젯을 사용하되 저장되는 값은 같은 검증 규칙을 따른다.

Win32 본문 편집기 경계의 줄바꿈 정책은 CRLF이다. DB, CLI, import에서 온 문자열이 LF 또는 단독 CR을 포함하면 편집기에 표시하기 전에 CRLF로 정규화하고, Rich Edit에서 본문을 읽을 때도 CRLF 기준으로 읽어 앱 내부 `String`에 반영한다. 이 정규화는 Win32 편집기 경계에서만 수행하며, repository와 도메인 계층은 Rich Edit 타입이나 RTF 표현을 알지 않는다.

## 텍스트 파일 가져오기/내보내기 인코딩 정책

내부 저장 정책은 항상 UTF-8 plain text이다. SQLite `nodes.content`는 UTF-8 문자열로 유지하고 Rust 내부 문서 본문 타입도 `String`을 사용한다. 텍스트 인코딩 선택은 외부 `.txt` 파일 import/export 경계에서만 적용한다.

지원하는 외부 텍스트 인코딩은 다음과 같다.

- `자동 감지` import 전용
- `UTF-8`
- `UTF-8 BOM`
- `UTF-16 LE BOM`
- `UTF-16 BE BOM`
- `한국어 (EUC-KR/CP949)`
- `Windows-1252`

`한국어 (EUC-KR/CP949)`은 `encoding_rs`의 `EUC_KR` 정책을 사용한다. 이 구현은 WHATWG EUC-KR 디코더를 따르며 일반적인 Windows-949 확장 범위를 함께 다루는 호환 정책으로 취급한다.

import 기본값은 `자동 감지`이다. `자동 감지` 판별 순서는 다음과 같다.

1. UTF-8 BOM, UTF-16 LE BOM, UTF-16 BE BOM을 먼저 확인한다.
2. BOM이 없으면 `UTF-8`로 유효한지 확인한다.
3. UTF-8이 아니면 `한국어 (EUC-KR/CP949)`로 디코딩을 시도한다.
4. 그래도 실패하면 `Windows-1252`로 fallback한다.

export 기본값은 `UTF-8`이다. BOM이 있는 export 인코딩은 파일 앞에 해당 BOM을 포함한다. `자동 감지`는 import 전용이므로 export 인코딩으로 저장하지 않는다.

`파일 > 저장`은 현재 활성 탭의 문서 본문을 저장한다. 저장할 변경 사항이 없거나 저장 가능한 활성 탭이 없으면 문서 내용을 바꾸지 않는다.

`파일 > 가져오기...`는 현재 활성 편집 문서 탭에 외부 텍스트 파일 내용을 불러온다. 편집 가능한 활성 문서가 없으면 사용자에게 안내한다. import 전에 대상 탭이 dirty이면 저장/버림/취소 확인을 수행하고, 취소하면 import를 중단한다. 디코딩에 실패하면 DB와 편집기 내용을 변경하지 않는다. 성공한 import는 편집기에 내용을 반영하고 dirty 상태로 두며 DB에 즉시 저장하지 않는다.

`파일 > 내보내기...`는 현재 활성 문서 탭의 본문을 외부 텍스트 파일로 저장한다. 편집기에 dirty 내용이 있으면 DB에 저장된 내용이 아니라 현재 편집기 내용을 기준으로 export한다. 단일 문서 export는 모든 플랫폼에서 공통 export 전용 인코더를 사용해 BOM, legacy encoding, byte limit 정책을 Windows 메뉴 경로와 동일하게 적용한다. 문서 탭이 없으면 사용자에게 안내한다.

`파일 > 모든 문서 내보내기...`는 활성 문서 트리 전체를 사용자가 선택한 폴더 아래의 `.txt` 파일들로 저장한다. 휴지통 문서는 내보내지 않는다. 각 문서는 `<node_id> - <제목>.txt` 파일로 저장하고, 하위 문서는 같은 이름의 하위 폴더 아래에 배치해 트리 계층을 보존한다. 예를 들어 `Root` 아래 `Untitled`가 있으면 `1 - Root.txt`와 `1 - Root/2 - Untitled.txt`를 만든다. 파일/폴더 이름에 사용할 수 없는 문자는 `_`로 바꾸고, 긴 제목은 경로 구성 요소 안에서 잘라낸다. 전체 내보내기도 `내보낼 인코딩` 설정을 사용하며, 열려 있는 dirty 활성 문서 탭은 아직 DB에 저장하지 않았더라도 현재 편집 내용을 기준으로 내보낸다. 선택한 폴더에 같은 경로의 `.txt` 파일이 이미 있으면 해당 파일을 교체하고, 그 밖의 파일은 건드리지 않는다.

`파일 > 종료`는 메인 창 닫기 버튼과 같은 앱 종료 흐름을 실행한다. dirty 탭이 있으면 앱 종료 정책에 따라 순서대로 자동 저장을 시도하고, 저장 취소나 실패가 발생하면 종료를 중단한다.

사용자가 마지막으로 선택한 import/export 인코딩은 SQLite `settings` 테이블에 다음 key로 저장한다. 저장된 값이 잘못되었거나 export에 `자동 감지`가 저장되어 있으면 기본값으로 fallback한다.

- `text.import.encoding`
- `text.export.encoding`

dirty 탭이 활성 문서에서 다른 문서로 이동하거나, dirty 탭을 닫거나, 앱을 종료할 때는 먼저 자동 저장을 시도한다. 저장 충돌이 발생하면 기존 DB 내용을 자동으로 덮어쓰지 않고 저장 충돌 선택지를 표시한다. 사용자가 취소하거나 저장에 실패하면 대상 탭 전환, 트리/검색 선택 전환, 탭 닫기, 앱 종료를 중단하고 현재 편집 내용과 dirty 상태를 유지한다.

편집기 내용이 마지막으로 로드하거나 저장한 본문과 달라지면 해당 탭은 dirty 상태가 된다. dirty 상태는 저장/전환/닫기 정책에만 사용하며 탭 제목이나 창 제목에 `*`로 표시하지 않는다.

Win32 편집기와 Linux GTK4 편집기는 텍스트 변경 알림마다 전체 본문을 즉시 읽지 않는다. 사용자 입력으로 편집 컨트롤이 변경되면 활성 탭을 dirty로 표시하고, 편집 컨트롤의 본문이 탭 모델보다 최신이라는 pending sync 상태만 기록한다. 정확한 본문이 필요한 저장, export, 찾기/바꾸기, 탭 전환/닫기, 앱 종료, import 전 dirty 확인, 편집 컨트롤 재생성 같은 유스케이스 경계에서만 편집 컨트롤의 전체 본문을 읽어 탭 모델에 동기화한다. Pending sync 상태에서는 활성 편집 컨트롤의 본문이 현재 편집 내용의 기준이다.
프로그램이 편집기나 검색창 텍스트를 설정하는 동안 발생한 재진입 변경 알림은 사용자 입력으로 취급하지 않는다. 이 알림은 suppress 상태로 처리하고, 같은 UI 상태가 이미 처리 중이라 빌릴 수 없으면 시작/탭 전환 흐름의 사용자 오류로 표시하지 않고 무시한다.
편집 알림 경로에서는 창 제목이나 탭 컨트롤을 갱신하지 않고, dirty/pending sync 상태만 바꾼다.
한국어 IME 조합 중 Rich Edit가 보내는 변경 알림도 같은 pending sync 정책을 따른다. 조합 중에는 dirty/pending sync 상태만 기록하고 검색 하이라이트 제거와 caret 상태 표시 갱신 같은 UI 후처리는 조합 종료 시 한 번 수행한다. 저장, 탭 전환, 찾기/바꾸기처럼 정확한 본문이 필요한 명령은 기존 유스케이스 경계에서 pending sync를 먼저 동기화한다.
저장 완료나 문서 metadata 갱신처럼 열린 탭 개수가 바뀌지 않는 갱신은 탭 UI 항목을 전체 삭제/재삽입하지 않고 기존 항목을 제자리에서 갱신한다. Windows는 Win32 탭 컨트롤 항목을 갱신하고, Linux GTK4는 기존 `Notebook` page와 `TextBuffer`를 유지한 채 탭 라벨, 편집 가능 상태, 표시 설정을 갱신한다. Linux에서 모델 본문이 실제로 바뀐 탭은 suppress 상태로 기존 `TextBuffer` 내용만 교체한다. 전체 재구성은 탭 열기/닫기처럼 항목 개수가 실제로 바뀌는 경우에만 수행한다.

문서 로드 시점의 `updated_at`은 열린 탭 상태에 함께 보관한다. 저장 시 repository는 트랜잭션 안에서 다음 형태의 조건부 UPDATE를 실행한다.

```sql
UPDATE nodes
SET content = ?, updated_at = ?
WHERE id = ? AND updated_at = ?
```

영향 받은 row가 0이면 같은 문서가 로드 이후 다른 창이나 프로세스에서 먼저 변경된 것으로 보고 저장 충돌로 처리한다. 충돌 시 기존 DB 내용을 덮어쓰지 않으며 사용자에게 다음 선택지를 제공한다. Windows `MessageBoxW`의 Yes/No/Cancel 의미를 기준으로, Linux GTK 대화상자도 Yes는 다시 로드, No는 다른 새 문서로 저장, Cancel/닫기는 취소로 처리한다.

저장 성공 시 새 `updated_at`은 로드 당시 `updated_at`보다 뒤의 값이어야 한다. 현재 시각의 정밀도 때문에 같은 값이 될 수 있으면 repository 계층에서 최소 단위만큼 앞으로 보정해, 이후 저장 충돌 검사용 토큰이 실제 변경 뒤에도 그대로 남지 않게 한다.

- 다시 로드: DB의 최신 본문과 `updated_at`을 다시 읽고 현재 편집 내용을 버린다. 같은 문서가 다른 열린 탭 동기화 대상에 있으면 clean 탭은 최신 본문으로 갱신하고, dirty 탭은 사용자의 현재 편집 본문과 저장 충돌 기준 토큰을 유지한다.
- 내 내용을 다른 새 문서로 저장: 같은 부모 문서 아래에 `기존제목 (복구됨)`를 기본 이름으로 하는 새 문서를 만들고, 이름이 겹치면 기존 새 문서 생성 규칙처럼 번호를 붙인다. 현재 탭은 생성된 새 문서 탭으로 전환한다.
- 취소: 저장을 중단하고 현재 편집 내용과 dirty 상태를 유지한다.

저장 성공 시 현재 활성 탭의 본문과 `updated_at`을 같은 트랜잭션에서 갱신한다. 저장이 성공하면 해당 탭의 로드된 `updated_at`을 새 값으로 갱신하고 dirty 상태를 해제한다. 창 제목에는 별도의 저장 완료 시간을 표시하지 않는다.

저장 실패 시 사용자에게 저장 실패 메시지를 표시하고 dirty 상태와 편집기 내용을 유지한다.

## 자동 저장 및 비정상 종료 복구 기반 정책

Rich Edit 전환 이후에도 문서 본문 모델은 plain text `String`이다. 자동 저장 또는 비정상 종료 복구 기능이 추가되더라도 복구 데이터는 Rich Edit 컨트롤에서 읽은 plain text만 사용할 수 있으며, RTF 조각이나 글자별 서식 같은 Rich Edit 임시 표현을 저장하면 안 된다.

현재 저장 정책은 사용자의 명시적 저장, dirty 탭 전환/닫기/종료 전 자동 저장, 저장 충돌 선택지, 저장할 수 없는 변경 확인을 기준으로 한다. 새 자동 저장 기능은 이 정책을 우회해 DB의 `nodes.content`를 조용히 덮어쓰면 안 된다. 특히 `updated_at` 조건부 저장과 저장 충돌 처리 없이 dirty 탭 내용을 DB에 반영하는 구현은 금지한다.

이번 기반 구현의 범위는 자동 저장 설정값의 도메인 모델에 한정한다. 기본값은 자동 저장 꺼짐이며, 설정값은 향후 UI 또는 실험 기능이 명시적으로 활성화하기 전까지 저장 동작을 바꾸지 않는다.

향후 복구 파일을 만들 경우 다음 원칙을 따른다.

- 복구 파일은 DB와 별도 경로에 plain text로만 저장한다.
- 복구 파일 생성은 dirty 탭의 현재 편집 내용을 보존하기 위한 후보 생성일 뿐, 명시적 저장으로 간주하지 않는다.
- 앱 시작 시 복구 후보를 감지하더라도 기존 DB 내용을 자동으로 덮어쓰지 않고 사용자가 복구/무시를 선택해야 한다.
- 복구 후보와 DB의 `updated_at`이 충돌하면 기존 저장 충돌 정책처럼 사용자가 다시 로드, 다른 새 문서로 저장, 취소 중 선택할 수 있어야 한다.
- 복구 파일 삭제 실패는 사용자에게 안내하되, DB 저장 성공 여부와 혼동하지 않는다.

임시 복구 파일은 문서 본문을 평문으로 남기므로 개인정보와 민감 문서가 디스크, 백업, 동기화 폴더, 보안 소프트웨어 스캔 대상에 노출될 수 있다. 자동 저장이나 복구 파일 기능을 켤 때는 저장 위치, 보존 기간, 삭제 실패 처리, 사용자 고지와 끄기 옵션을 함께 설계해야 한다.

## 열린 탭과 문서 전환 정책

문서 노드를 선택하면 오른쪽 영역에 해당 문서를 탭으로 연다. 이미 열린 문서이면 새 탭을 만들지 않고 기존 탭을 활성화한다. 검색 결과 문서도 같은 규칙으로 탭을 연다.

탭을 클릭해 활성화하면 편집 컨트롤의 현재 내용과 스크롤 위치는 이전 활성 탭 상태에 저장되고, 새 활성 탭의 내용과 스크롤 위치가 편집기에 복원된다. 가능한 경우 왼쪽 트리 선택도 활성 탭의 노드로 동기화한다. 현재 보기에서 해당 노드가 보이지 않으면 선택 동기화 실패를 사용자 오류로 표시하지 않는다.

새로 열린 탭, 다시 로드된 탭, 가져오기/버림처럼 본문이 새 세션으로 교체된 탭은 첫 줄에서 표시를 시작한다.

탭 전환이나 트리/검색 결과에서 다른 문서 선택은 현재 활성 탭이 dirty이면 먼저 자동 저장한다. 저장이 성공하면 대상 문서 탭으로 전환하고, 저장 충돌에서 취소를 선택하거나 저장에 실패하면 전환을 중단한다. 저장할 수 없는 읽기 전용 dirty 탭은 변경 내용을 버리고 계속할지 확인하며, 취소하면 전환을 중단한다.

탭 닫기는 `파일 > 탭 닫기` 또는 `Ctrl+W`로 수행한다. dirty 탭을 닫으려 하면 먼저 자동 저장하고, 저장이 성공하면 탭을 닫는다. 저장 충돌에서 취소를 선택하거나 저장에 실패하면 탭 닫기를 중단한다. 저장할 수 없는 읽기 전용 dirty 탭은 변경 내용을 버리고 닫을지 확인한다. 미저장 변경 확인 대화상자는 Windows Yes/No/Cancel 의미를 기준으로 Yes=저장, No=버림, Cancel/닫기=중단으로 처리한다.

각 탭에는 닫기 버튼을 표시한다. 닫기 버튼을 클릭하면 해당 탭을 닫는다. 비활성 탭의 닫기 버튼을 클릭하면 먼저 현재 활성 탭 전환 정책을 따른 뒤 대상 탭 닫기 정책을 적용한다. 전환 또는 닫기 과정에서 저장이 취소되거나 실패하면 닫기를 중단한다.

탭을 마우스로 잡아 좌우로 드래그하면 열린 탭 목록 안에서 표시 순서를 바꾼다. 순서 변경은 문서 본문, dirty 상태, 저장 충돌 토큰을 바꾸지 않으며, 현재 활성 문서는 같은 탭을 계속 가리키도록 유지한다.

마지막 탭을 닫으면 편집기는 빈 읽기 전용 상태가 된다.

트리를 다시 채우는 생성, 이름 변경, 이동, 정렬, 검색 갱신, 보기 전환은 가능한 한 열린 탭과 dirty 내용을 보존한다. 이름 변경이나 이동처럼 앱 내부 작업이 문서 메타데이터의 `updated_at`을 바꾸는 경우에는 dirty 탭의 현재 본문은 유지하되 저장 충돌 토큰을 새 값으로 갱신한다.

삭제, 영구 삭제처럼 특정 열린 탭의 대상 노드를 더 이상 유효하지 않게 만드는 작업은 대상 노드와 하위 노드에 해당하는 dirty 탭을 먼저 확인한다. 저장이 성공하면 작업을 계속하고, 버림은 해당 탭 변경을 버리고 작업을 계속하며, 취소는 작업을 중단한다. 작업이 완료되면 삭제 또는 영구 삭제된 노드의 탭은 닫는다.

앱 종료 시 dirty 탭이 하나 이상 있으면 각 dirty 탭을 순서대로 자동 저장한다. 모든 저장이 성공하면 종료하고, 저장 충돌에서 취소를 선택하거나 저장에 실패하면 종료를 중단한다. 저장할 수 없는 읽기 전용 dirty 탭은 변경 내용을 버리고 종료를 계속할지 확인한다.

## 검색 정책

검색은 모든 활성 문서 노드의 `title`과 `content`를 대상으로 한다.

검색어는 앞뒤 공백을 제거한 뒤 적용한다. 빈 검색어는 검색 보기를 닫고 일반 활성 트리 표시로 돌아간다.

현재 검색은 제목에는 SQLite `LIKE` 부분 일치를 적용하고, 본문에는 가능한 경우 SQLite FTS trigram `MATCH`를 적용한다. FTS를 사용할 수 없는 짧은 검색어는 제목 결과가 제한 개수를 채우지 못할 때만 제한된 수의 본문 `LIKE` 후보를 보충한다. 사용자가 입력한 `%`, `_`, `^` 문자는 와일드카드가 아니라 일반 문자로 검색되도록 이스케이프한다.

대소문자 구분은 SQLite 기본 `LIKE` 동작을 따른다. 즉 ASCII 영문은 기본적으로 대소문자를 구분하지 않고, 그 밖의 문자는 SQLite 빌드와 collation 기본 동작을 따른다.

`deleted_at IS NOT NULL`인 삭제 문서는 기본 검색 결과에서 제외한다.

검색 결과는 UI 멈춤을 줄이기 위해 상위 200개로 제한한다. 결과 항목에는 문서 제목과 부모 문서 제목을 표시한다.

검색 결과 문서를 선택하면 일반 트리 선택과 같은 탭 열기 정책을 적용한다. 현재 활성 탭이 dirty이면 먼저 자동 저장하고, 저장이 성공한 뒤 검색 결과 문서를 새 탭으로 열거나 기존 탭으로 전환한다.

검색어 변경은 Windows 동작을 기준으로 처리한다. 입력 문자열이 기존 검색어와 같으면 pending 검색 갱신을 취소하고 아무 작업도 하지 않는다. trim 결과만 같으면 트리 갱신 없이 내부 검색어 문자열만 갱신한다. 빈 검색으로 돌아가거나 현재 보이는 제목 검색 결과를 더 좁히는 입력은 즉시 갱신한다. DB 재검색이 필요한 비어 있지 않은 검색어 변경만 180ms debounce한다. 현재 결과에서 좁히는 최적화는 본문 매치가 포함되지 않은 검색 결과이고, 검색어 길이 변화가 SQLite FTS trigram 본문 검색 적용 범위를 바꾸지 않을 때만 사용한다.

## 텍스트 바꾸기 정책

`편집 > 찾기...`와 `Ctrl+F`는 현재 활성화된 편집 가능한 문서 탭의 본문만 대상으로 텍스트 찾기 UI를 연다. `편집 > 바꾸기...`와 `Ctrl+H`는 같은 대상에 대해 텍스트 바꾸기 UI를 연다. 같은 종류의 찾기/바꾸기 UI가 이미 열려 있으면 새 창을 만들지 않고 기존 창에 포커스를 주며, 다른 종류의 UI가 열려 있으면 기존 창을 닫고 요청한 종류를 새로 연다. 휴지통 문서처럼 읽기 전용 탭이거나, 편집 가능한 활성 문서 탭이 없으면 사용자에게 안내하고 동작하지 않는다. 트리 선택이 활성 탭과 달라도 찾기/바꾸기는 활성 편집 탭을 기준으로 동작하며, UI가 열린 뒤 탭 상태가 바뀐 경우에도 각 실행 시점에 편집 가능 상태를 다시 확인한다.

바꾸기는 DB를 직접 수정하지 않고 현재 편집기 본문과 열린 탭 상태만 갱신한다. 변경된 본문은 기존 dirty 추적과 저장 정책을 따른다. 한 번 바꾸기는 현재 선택된 match 범위만 편집기에서 교체하고, 모두 바꾸기는 계산된 전체 본문을 적용한다. 본문이 실제로 바뀌면 탭 라벨과 창 제목의 dirty 표시를 즉시 갱신한다.

찾을 문자열과 바꿀 문자열을 입력할 수 있으며, Windows 공용 찾기/바꾸기 대화상자의 1024 UTF-16 code unit 버퍼에 맞춰 각 입력값은 종료 null을 제외한 1023 UTF-16 code unit까지 허용한다. 찾을 문자열이 비어 있으면 사용자에게 안내하고 바꾸기를 실행하지 않는다. 찾기/바꾸기 입력칸에서 Enter를 누르면 대화상자의 기본 실행인 다음 찾기를 수행한다. 다음 찾기는 현재 selection 또는 caret 뒤에서 시작하고, 문서 끝까지 일치 항목이 없으면 문서 처음부터 다시 찾는다. 한 번 바꾸기는 현재 selection이 찾을 문자열과 정확히 일치할 때만 치환하고, 치환 후 다음 일치 항목을 같은 래핑 규칙으로 선택한다. 다음 일치 항목이 없으면 치환된 텍스트 뒤에 caret을 둔다. 찾기/바꾸기 대화상자에서 실행한 다음 찾기/바꾸기는 편집기 selection과 스크롤 위치를 갱신하지만, 대화상자 포커스를 편집기로 강제로 옮기지 않는다. 모두 바꾸기는 현재 본문 전체에서 겹치지 않는 문자열 일치를 바꾸고, 실행 후 바꾼 개수를 사용자에게 알려준다.

Win32 본문 편집기는 표시 텍스트의 UTF-8 byte range와 컨트롤 선택 offset을 명시적으로 변환한다. 도메인 검색/바꾸기 결과는 Rust `String`의 UTF-8 byte range로 유지한다. 본문 문자열은 CRLF 기준 plain text로 읽고 저장하지만, Rich Edit 선택 API의 원시 character position은 CRLF 쌍을 내부 줄바꿈 1개로 취급한다. 따라서 Win32 경계 변환은 UTF-8 byte range와 UTF-16 코드 유닛 offset을 왕복하되, `EM_EXGETSEL`/`EM_EXSETSEL`에 전달하는 offset에서는 CRLF 쌍을 1개 위치로 접고, 앱 내부 문자열에는 CRLF 전체를 유지한다.

찾기/바꾸기에서 현재 찾은 항목은 Rich Edit selection으로 표시하고, 컨트롤이 plain text 모드에서도 범위별 배경색 서식을 허용하는 경우에만 임시 배경색을 보조로 적용한다. 이 표시는 Win32 편집기 경계의 transient view state이며 문서 본문 모델, 저장 내용, dirty 판단에 포함하지 않는다. 저장, 동기화, 사용자 편집, 탭/문서 전환, 검색어 변경, 찾기 닫기 경계에서는 임시 배경색을 제거하고, 다시 찾기를 실행하면 현재 테마에 맞는 읽기 가능한 색으로 현재 일치 항목만 다시 표시한다.

## 단축키 정책

- `Ctrl+S`: 현재 활성 탭의 문서 본문을 저장한다.
- `Ctrl+W`: 현재 활성 탭을 닫는다.
- `Ctrl+A`: 편집 영역에 포커스가 있을 때 현재 활성 탭의 본문 전체를 선택한다.
- `Enter`: 트리뷰에 포커스가 있을 때 선택된 문서의 형제로 새 문서를 만든다. 선택된 문서가 루트이면 루트 아래에 만든다. 기존 `Ctrl+N`도 같은 동작을 수행한다.
- `Ctrl+Enter`: 트리뷰에 포커스가 있을 때 선택된 문서의 하위 문서를 만든다.
- `Ctrl+F`: 현재 활성화된 편집 가능한 문서 본문에 대해 텍스트 찾기 UI를 연다.
- `Ctrl+H`: 현재 활성화된 편집 가능한 문서 본문에 대해 텍스트 바꾸기 UI를 연다.
- `F2`: 트리뷰에 포커스가 있을 때 현재 선택된 노드 이름 변경을 시작한다.
- `Delete`: 활성 트리에서 트리뷰에 포커스가 있을 때 현재 선택된 노드를 soft delete한다. 휴지통 보기에서는 현재 선택된 노드를 확인 후 영구 삭제한다. 키보드 단축키는 메뉴의 삭제/영구 삭제와 같은 확인, dirty 탭 처리, 트리 갱신 흐름을 사용하고 완료 후 트리뷰 포커스를 유지한다.
- `Ctrl+Up` / `Ctrl+Down`: 트리뷰에 포커스가 있을 때 선택된 노드를 같은 부모 아래에서 한 칸 위 또는 아래로 이동한다.
- `Right` / `Left` / `+` / `-` / `*`: 트리뷰에 포커스가 있을 때 Win32 TreeView 기준으로 접기/펼치기와 부모/첫 자식 이동을 수행한다. `Right`는 접힌 선택 노드를 펼치고 이미 펼쳐져 있으면 첫 자식을 선택한다. `Left`는 펼쳐진 선택 노드를 접고 이미 접혀 있거나 자식이 없으면 부모를 선택한다. `+`와 `-`는 선택 노드를 펼치거나 접고, `*`는 선택 노드의 하위 트리를 모두 펼친다. 검색 결과 트리도 처음 표시될 때는 Windows처럼 전체 펼침으로 시작하지만 이후 사용자 접기/펼치기 상태를 반영한다.
- `Context Menu` / `Shift+F10`: 트리뷰 또는 편집 영역에 포커스가 있을 때 해당 영역의 우클릭 메뉴를 연다.

## UI 표시 정책

메인 메뉴의 최상위 구조는 `파일`, `편집`, `문서`, `보기`, `도움말` 순서로 둔다. 영어 UI에서는 같은 구조를 `File`, `Edit`, `Document`, `View`, `Help`로 표시한다. 문서 파일 입출력과 탭/앱 종료는 `파일`, 텍스트 편집과 찾기/바꾸기는 `편집`, 트리 문서 생성/이름 변경/이동/삭제/복원은 `문서`, 보기 전환과 테마/언어/편집기 표시 설정은 `보기`, 앱 정보는 `도움말`에 둔다.

편집 메뉴의 실행 취소, 잘라내기, 복사, 붙여넣기, 삭제, 전체 선택, 찾기/바꾸기 활성화 상태는 활성 탭 존재 여부, 편집 가능 여부, 현재 편집기 selection, undo 가능 여부, 현재 편집기 텍스트 길이를 기준으로 계산한다. pending sync나 플랫폼별 모델 캐시가 있더라도 메뉴 상태 판단에서는 화면의 live 편집 컨트롤 상태를 우선한다.

메인 창은 좌우 분할 레이아웃을 사용한다. 왼쪽은 `nodes` 테이블의 문서 계층을 표시하는 트리 영역이고, 오른쪽은 열린 문서 탭 바와 텍스트 편집 영역이다. 기본 분할 폭은 왼쪽 280px이다. 96 DPI 기준으로 검색창은 24px 높이와 6px inset, splitter는 4px 폭, 탭 바는 28px 높이, caret 상태줄은 22px 높이와 좌우 8px inset을 기준으로 배치한다.

Windows GUI는 창을 만들기 전에 프로세스 DPI awareness를 Per-Monitor v1 우선, Per-Monitor V2 fallback 순서로 설정한다. 고배율 화면에서 Windows의 비트맵 확대 때문에 글자가 흐려지지 않도록 편집기 글꼴은 현재 창 DPI를 기준으로 생성하고, 96 DPI 기준 픽셀 배치 상수는 `UiScale`을 통해 현재 창 DPI에 맞춘다. 저장된 창 geometry와 splitter 위치는 사용자가 정한 실제 창 상태로 취급해 DPI 비율 값을 별도로 저장하거나 시작 시점마다 재보정하지 않는다. 단, 저장된 창 위치가 없는 최초 기본 배치는 시작 DPI 기준으로 기본 창 크기와 splitter 폭을 한 번 스케일한다.

Win32 native size/move 루프 중에는 모니터 경계 이동의 끊김을 줄이기 위해 `WM_ENTERSIZEMOVE`부터 `WM_EXITSIZEMOVE`까지 별도 상태를 둔다. 이 구간에 들어온 `WM_DPICHANGED`는 Windows suggested rectangle을 즉시 `SetWindowPos`로 적용하지 않고 pending으로 표시한다. size/move 종료 뒤 실제 현재 창 DPI와 client rect 기준으로 편집기 글꼴, 테마, 레이아웃을 한 번만 갱신한다. 검색 debounce 타이머도 size/move 중에는 실행하지 않고 종료 뒤 처리해 DB 검색과 트리 재구성이 창 이동을 막지 않게 한다.

Windows GUI 실행 파일과 메인 창은 프로젝트 루트의 `icon.ico`를 Windows 리소스 ID 1번으로 포함해 같은 애플리케이션 아이콘을 사용한다.

Linux GTK4 application id는 `io.github.edgarp9.j3TreeText`다. GTK default icon name과 window `icon_name`도 같은 값을 사용한다. 직접 실행 시에는 실행 파일 폴더 또는 현재 작업 폴더에서 `icon.svg`, `icon.png` 순서로 런타임 아이콘 파일을 찾아 커스텀 타이틀바와 GDK toplevel icon list에 적용한다. `--install`은 XDG user data 경로 아래에 desktop entry와 hicolor 앱 아이콘을 설치하며, 같은 내용을 다시 쓰지 않는 idempotent 동작이다. application id에 ASCII 대문자가 있으므로 KDE/Plasma fallback을 위해 lowercase alias desktop entry/icon을 함께 설치하고, `--uninstall`은 현재 application id, lowercase alias, legacy id의 desktop entry/icon을 제거한다.

`도움말 > j3TreeText 정보`는 제목이 `About j3TreeText`인 작은 modal about 창을 연다. 상단에는 Cargo 패키지 버전 기반의 `j3TreeText <version>` 라벨을 표시하고, 본문에는 실행 파일 옆 `about.txt` 또는 빌드에 포함된 fallback 내용을 읽기 전용 스크롤 텍스트 영역에 원문 형식대로 표시한다. 하단 왼쪽에는 `https://github.com/edgarp9` URL 버튼 또는 링크를, 하단 오른쪽에는 `OK` 버튼을 둔다. URL을 클릭하면 기본 브라우저로 열고, 실패하면 기존 데스크톱 UI 오류 표시 경로로 사용자에게 알린다. Windows GUI에서는 about 창을 owner인 메인 창의 중앙에 배치하되, 가능한 경우 현재 모니터 작업 영역 밖으로 벗어나지 않게 보정한다.

의존성 또는 포함 리소스가 바뀌면 `python tools/generate_third_party_notices.py`로 `THIRD_PARTY_NOTICES.txt`를 갱신하고, About 창의 고지 파일 안내가 실제 배포 파일과 일치하는지 확인한다. 배포 산출물에는 `LICENSE`, `about.txt`, `THIRD_PARTY_NOTICES.txt`를 함께 포함하고, 바이너리 공개 배포 시 해당 빌드의 대응 소스를 제공한다.

트리 표시 순서는 같은 부모 아래에서 `sort_order`, `title`, `id` 기준으로 정렬한다. 활성 트리에서는 문서 제목만 표시하고, 휴지통에서는 한국어 UI에서 `[삭제됨] 제목`, 영어 UI에서 `[Deleted] 제목` 형식으로 표시한다.

문서 노드를 선택하면 오른쪽 편집 영역에 탭을 열고 `content`를 표시한다. 활성 트리와 검색 결과의 문서 탭은 편집 가능 상태로 둔다. 본문 수정 내용은 탭별 dirty 상태로 추적하며, 저장 명령이 실행되면 현재 활성 탭만 DB에 저장한다. 저장 성공 후 탭은 dirty 상태와 저장 충돌 토큰만 갱신하고, 별도의 마지막 저장 시각 UI 상태는 유지하지 않는다.

각 문서 탭의 오른쪽에는 닫기 버튼을 표시한다. 닫기 버튼은 메뉴의 `파일 > 탭 닫기`와 같은 저장/충돌/취소 흐름을 사용한다.

문서 탭은 탭 바 안에서 좌우 드래그로 재정렬할 수 있다. 재정렬 결과는 현재 실행 중인 열린 탭 표시 순서에만 적용하며 별도 설정으로 영구 저장하지 않는다.

휴지통의 문서 탭은 읽기 전용으로 열린다. 휴지통 탭은 dirty가 되지 않고 저장 대상에서 제외된다. 복원된 문서의 탭은 활성 문서 메타데이터와 동기화되어 다시 편집 가능한 탭이 될 수 있다.

Windows 본문 편집 영역은 Win32 Rich Edit 컨트롤을 사용하고, Linux 본문 편집 영역은 GTK4 `TextView`를 사용한다. 앱의 문서 본문은 두 플랫폼 모두 plain text로 유지한다.

Rich Edit 적용 범위는 Win32 UI 경계에 한정한다. `platform/win32`는 Rich Edit 모듈 초기화, 컨트롤 클래스 선택, multiline/scroll/readonly/word wrap 스타일, 텍스트 읽기와 쓰기, 선택/스크롤 복원 메시지 차이를 흡수한다. GTK4 `TextView` 적용 범위도 Linux UI 경계에 한정한다. `app`, `domain`, repository 계층은 Rich Edit, GTK 위젯 타입, RTF 표현을 알지 않는다.

Rich Edit를 사용할 때도 본문 로드/저장/import/export는 UTF-8 plain text 정책을 따른다. 컨트롤에 표시할 때는 UTF-8 `String`을 CRLF 기준 UTF-16 plain text로 변환하고, 저장/export/찾기/바꾸기 경계에서 컨트롤의 UTF-16 텍스트를 CRLF 기준 UTF-8 `String`으로 동기화한다. 변환 유틸은 UTF-8 byte range와 UTF-16 코드 유닛 offset 사이를 명시적으로 왕복해야 한다. 단, Rich Edit 선택 API는 CRLF를 내부 줄바꿈 1개 위치로 세므로 선택 offset 변환에서는 CRLF 쌍을 하나로 접고, 저장되는 plain text에는 CRLF 쌍을 그대로 유지한다.

텍스트 파일 import의 원본 파일 크기는 공통 파일 I/O 계층에서 16MiB byte 한도로 검사한다. import 후 편집기에 표시할 본문은 CRLF 정규화 뒤 Windows Rich Edit 기준과 같은 64MiB UTF-16 코드 유닛 표시 한도로 검사한다. Linux GTK4 백엔드도 이 Windows 기준 한도를 따라야 하며, UTF-8 byte 길이만으로 정규화된 본문을 거부하지 않는다. 사용자가 편집기에 직접 입력하거나 붙여넣는 본문도 같은 Windows 편집기 UTF-16 코드 유닛 한도를 넘지 않아야 한다.

문서 탭별 편집기 view state는 첫 visible line, caret 위치, selection start/end를 저장한다. selection과 caret offset은 Win32 편집기 경계와 Linux GTK4 편집기 경계 모두에서 UTF-16 코드 유닛 기준으로 관리하고, 탭을 다시 표시할 때 현재 본문 길이와 가능한 line count 안으로 보정한다. 이 복원 과정은 사용자 편집이 아니므로 dirty 상태를 만들지 않는다.

본문 편집기 하단에는 현재 caret 위치를 표시한다. 한국어 UI에서는 `줄 N, 열 M`, 영어 UI에서는 `Ln N, Col M` 형식으로 표시한다. 줄 번호와 열 번호는 1부터 시작한다. Windows 줄/열은 Rich Edit가 현재 자동 줄바꿈 설정과 컨트롤 폭으로 배치한 표시 줄 기준이며, 자동 줄바꿈 토글이나 DPI/창 크기 변경 뒤에는 같은 caret offset이라도 표시 줄이 바뀔 수 있다. 열은 해당 표시 줄 시작 Rich Edit offset부터 caret offset까지의 UTF-16 코드 유닛 수에 1을 더한 값으로 계산한다. 따라서 BMP 문자는 1열, surrogate pair 문자는 2열을 차지하고, CRLF 줄바꿈은 Rich Edit 선택 offset 정책처럼 하나의 문단 구분 위치로 접는다. Linux GTK4도 `TextView` display line 시작 위치와 caret 사이를 UTF-16 코드 유닛 기준으로 계산해 Windows 상태줄 기준을 따른다. 이 상태 표시는 편집 가능 탭과 읽기 전용 탭 모두에서 동작하며 문서 본문이나 dirty 상태를 바꾸지 않는다.

GUI 기본 배치는 Windows Win32 레이아웃 수치를 기준으로 한다. 왼쪽 검색창은 문서 트리 영역 안에서 6px inset과 24px 높이를 사용하고, 트리 자체는 왼쪽 영역 전체 폭을 채운다. splitter 폭은 4px, 왼쪽 최소 폭은 80px, 오른쪽 편집 영역 최소 폭은 160px이다. 하단 caret 상태줄은 22px 높이와 좌우 8px inset을 사용한다. Linux GTK4 백엔드는 플랫폼 기본 위젯을 쓰더라도 이 fixed metric이 사용자 경험을 바꾸지 않도록 가능한 범위에서 height request와 margin으로 보정한다.

Rich Edit 경계 회귀 테스트는 빈 문자열, ASCII 단일 줄, 한글, surrogate pair 이모지, CRLF 여러 줄, UTF-8 문자 경계에 맞지 않는 선택 범위를 포함한다. 저장소 계층 테스트는 Rich Edit/RTF처럼 보이는 문자열도 해석하지 않고 UTF-8 plain text 본문으로 그대로 저장되는 정책을 고정한다.

편집 영역의 글꼴은 메뉴의 `보기 > 편집기 글꼴...`에서 변경한다. Windows는 표준 `ChooseFontW` 대화상자를, Linux는 GTK4 `FontChooserDialog`를 사용한다. 대화상자는 플랫폼 표준 선택/취소 버튼을 사용하고, 시스템에 설치된 화면 글꼴을 대상으로 하며, family/style/size 선택 범위만 노출하고, 취소하면 기존 편집기 글꼴 설정을 유지한다. 적용 대상은 오른쪽 문서 편집 컨트롤이며 트리뷰, 검색창, 메뉴 글꼴은 변경하지 않는다.

편집기 글꼴 설정 키는 다음과 같다.

- `editor.font.family`
- `editor.font.size_pt`

기본값은 `Consolas`, `10pt`이다. 글꼴 크기는 6pt 이상 72pt 이하만 유효한 저장값으로 인정한다. 저장된 글꼴 이름이 비어 있거나 크기 문자열 파싱에 실패하거나 범위를 벗어나면 기본값으로 fallback한다. Windows 글꼴 대화상자의 픽셀/DPI 변환 때문에 반환값이나 다음 초기 표시가 정수 pt보다 작아지더라도, 사용자가 선택한 정수 pt 설정은 다시 대화상자를 열 때 같은 정수로 표시되어야 한다. 플랫폼 대화상자가 1/10pt 또는 fractional size를 반환하면 Windows `ChooseFontW` 처리처럼 정수 pt로 올림한 뒤 저장 범위로 clamp한다. 저장된 글꼴이 현재 시스템에서 찾을 수 없으면 앱 시작 또는 적용 시 기본 편집기 글꼴로 fallback하고, fallback 결과를 UI 설정에 저장한다. 새로 선택한 글꼴 적용 중 fallback이 발생하면 사용자에게 기본 글꼴이 사용되었음을 알린다.

앱 시작 시 repository 계층이 설정을 읽고 플랫폼별 UI 생성 뒤 편집 컨트롤에 복원한다. Windows 편집 컨트롤에는 선택된 `HFONT`를 적용하고, Linux 편집 컨트롤에는 GTK CSS로 글꼴 family와 point size를 적용한다. 글꼴 변경은 문서 본문 변경이 아니므로 열린 문서 탭의 dirty 상태나 저장 충돌 토큰에 영향을 주지 않는다. 글꼴 변경 전 현재 편집기 본문은 활성 탭 상태에 먼저 보관하고, 글꼴 적용 뒤 같은 본문과 가능한 selection/caret 상태를 유지한다.

편집 영역 자동 줄바꿈은 메뉴의 `보기 > 자동 줄바꿈`에서 켜고 끈다. 설정 키는 `editor.word_wrap`이고 저장값은 `"true"` 또는 `"false"` 문자열이다. 기본값은 켜짐(`true`)이며, 저장된 값이 잘못되면 켜짐으로 fallback한다. 앱 시작 시 repository 계층이 설정을 읽어 오른쪽 문서 편집 컨트롤에 복원한다. 자동 줄바꿈 변경은 문서 본문 변경이 아니므로 열린 문서 탭의 dirty 상태나 저장 충돌 토큰에 영향을 주지 않고, 가능한 경우 현재 편집기 selection/caret과 undo 상태를 유지한 채 표시 속성만 바꾼다. 자동 줄바꿈을 끄면 편집 영역은 가로 스크롤을 사용하고, 켜면 편집 영역 폭에 맞춰 줄을 감는다.

테마는 메뉴의 `보기 > 테마` 하위 메뉴에서 선택한다. 기본값은 `밝게`이고 설정 키는 `appearance.theme`이다. 저장값은 `light`, `classic-dark`, `sepia-teal`, `graphite`, `forest`, `steel-blue` 중 하나이며, 저장된 값이 잘못되면 `light`로 fallback한다. `세피아`, `그래파이트`, `숲`, `스틸 블루`는 참고 스크린샷의 색감을 바탕으로 한 다크 계열 팔레트이다. 기존 DB의 `appearance.dark_theme=true`는 새 키가 없을 때 `classic-dark`로, `false`는 `light`로 읽는다. 앱 시작 시 repository 계층이 설정을 읽고 플랫폼별 UI 생성 뒤 창 배경, 트리, 검색 입력, 탭, 편집 컨트롤에 가능한 범위에서 색상을 적용한다. Win32 컨트롤이 부분 다시그리기 중 재진입 색상 질의를 보내더라도 마지막 적용 테마의 색상으로 배경과 텍스트를 그려야 한다. Rich Edit 본문 편집기는 일반 컨트롤 색상 질의와 별도로 배경색과 글자색을 적용하며, Linux GTK4 백엔드는 CSS provider로 같은 팔레트의 배경색과 글자색을 적용한다. Linux GTK4의 `밝게` 테마는 시스템 GTK 다크 테마를 사용 중이어도 메뉴바, 검색 입력, 트리, 탭, 상태줄, 편집기 주변 표면에 명시적인 밝은 배경과 검은 글자색을 적용해 다크 배경을 상속하지 않는다. 밝은 테마로 돌아가면 각 플랫폼의 기본 색상과 일관된 밝은 팔레트로 복원한다. 테마 변경은 문서 본문 변경이 아니므로 열린 문서 탭의 dirty 상태나 저장 충돌 토큰에 영향을 주지 않고, 가능한 경우 편집 컨트롤을 재생성하지 않는다.

UI 언어는 메뉴의 `보기 > 언어` 하위 메뉴에서 `한국어` 또는 `English`를 선택한다. 기본값은 English이고 설정 키는 `ui.language`이다. 저장값은 `ko` 또는 `en`이며, 저장된 값이 잘못되면 `en`으로 fallback한다. 언어 변경은 메뉴, 우클릭 메뉴, 검색 입력 힌트, 창 제목의 보기 상태 접미사, 휴지통/검색 결과 표시 보조 문구, 파일/글꼴/확인/about 대화상자의 앱 제공 문구에 즉시 적용한다. 문서 제목과 문서 본문처럼 사용자가 저장한 데이터는 번역하지 않는다.

문서 CRUD 진입점은 메뉴의 `문서` 항목이다. `새 문서`는 선택 문서의 형제로, `새 하위 문서`는 선택 문서의 자식으로 생성한 뒤 바로 트리뷰 라벨 편집을 시작한다. `이름 변경`은 트리뷰 라벨 편집으로 처리한다. Windows에서는 Win32 TreeView 라벨 편집 알림을 사용하고, Linux에서는 선택 행 안의 GTK `Entry`를 사용한다. 라벨 편집 완료 시 먼저 현재 편집기 본문과 열린 탭 metadata를 Windows의 새로고침 전 경계와 같이 동기화한 뒤 이름 변경을 커밋한다. 이름 변경 성공 후에는 active document metadata 기준으로 열린 탭 제목과 dirty 저장 충돌 토큰을 갱신하고 트리를 다시 채운다. Windows에서는 Win32 TreeView 알림 처리가 끝난 뒤 별도 UI 메시지로 트리를 다시 채우며, 이때 TreeView 컨트롤의 자체 라벨 텍스트 반영은 거부해 재구성된 항목 핸들에 이전 편집 텍스트가 덮어써지지 않게 한다. 생성/이름 변경/삭제/이동 후 트리는 활성 노드 목록으로 다시 구성한다. 가능하면 생성/변경/이동 대상 또는 삭제 대상의 부모를 선택한다.

휴지통 진입점은 메뉴의 `보기 > 휴지통` 항목이다. `보기 > 문서 트리`로 일반 트리 보기로 돌아간다. 휴지통 또는 검색 결과에서 일반 활성 트리로 돌아갈 때는 active document metadata를 다시 읽고 열린 탭을 동기화한다. 현재 활성 탭 본문은 편집기 상태 보존을 위해 다시 읽지 않고, 오래된 clean 비활성 탭 본문은 최신 active document 본문으로 갱신한다. active document에서 사라진 열린 탭은 본문을 보존하되 읽기 전용으로 바꾸고 저장 대상에서 제외한다. 휴지통에서 선택한 노드는 `문서 > 복원`으로 복원하고 `문서 > 영구 삭제`로 영구 삭제한다.

메뉴는 현재 보기와 선택 상태를 반영한다. 활성 트리 일반 보기에서는 새 문서/새 하위 문서를 사용할 수 있고, 선택된 노드가 있을 때 이름 변경을 사용할 수 있으며, 루트가 아닌 선택 노드만 휴지통으로 이동할 수 있다. 선택 노드가 같은 부모 아래에서 위나 아래로 이동 가능한 위치에 있으면 `위로` 또는 `아래로`를 사용할 수 있다. 검색 결과 보기에서는 문서 생성/이름 변경/이동/휴지통 이동/복원/영구 삭제 메뉴를 사용할 수 없다. 휴지통 보기에서는 선택된 삭제 노드가 있을 때만 복원과 영구 삭제를 사용할 수 있다. `보기 > 문서 트리`는 활성 트리와 검색 결과 보기에서 checked 상태이고, `보기 > 휴지통`은 휴지통 보기에서 checked 상태이다. `보기 > 테마`와 `보기 > 언어`는 현재 선택된 항목만 checked 상태로 반영한다.

왼쪽 트리 영역의 우클릭 메뉴는 현재 보기와 선택 상태를 반영해 문서 생성, 이름 변경, 위/아래 이동, 휴지통 이동, 복원, 영구 삭제 명령을 표시한다. 트리 항목 위에서 우클릭하면 먼저 해당 항목을 선택한 뒤 명령 상태를 계산한다. 우클릭 메뉴에서 실행한 명령은 같은 우클릭 제스처로 메뉴를 다시 열지 않는다. 오른쪽 편집 영역의 우클릭 메뉴는 실행 취소, 잘라내기, 복사, 붙여넣기, 삭제, 전체 선택 명령을 제공한다. 편집기 우클릭은 플랫폼 기본 텍스트 위젯 메뉴가 아니라 앱의 편집기 우클릭 메뉴 계약만 표시한다. 편집 가능한 활성 문서 탭에서만 본문 변경 명령을 사용할 수 있고, 읽기 전용 문서에서는 복사와 전체 선택처럼 본문을 바꾸지 않는 명령만 허용한다. 키보드 컨텍스트 메뉴는 Windows의 현재 커서 위치 기준을 따른다. Linux/Wayland처럼 전역 커서 좌표를 안정적으로 읽을 수 없는 환경에서는 앱 내부에서 마지막으로 관측한 해당 트리/편집기 포인터 좌표를 우선 사용하고, 없으면 포커스 위젯 또는 선택 행 기준으로 표시한다.

## UI 상태 저장 정책

UI 상태는 SQLite `settings(key TEXT PRIMARY KEY, value TEXT NOT NULL)` 테이블에 문자열 값으로 저장한다. 저장 대상 키는 다음과 같다.

- `window.x`
- `window.y`
- `window.width`
- `window.height`
- `splitter.left_width`
- `selection.node_id`
- `editor.font.family`
- `editor.font.size_pt`
- `editor.word_wrap`
- `text.import.encoding`
- `text.export.encoding`
- `appearance.theme`
- `appearance.dark_theme`
- `ui.language`
- `autosave.enabled`
- `autosave.interval_seconds`

자동 저장 설정 키는 현재 도메인 모델과 설정 저장소 왕복만 지원한다. `autosave.enabled` 기본값은 `"false"`이고, `autosave.interval_seconds` 기본값은 `120`초이다. 저장된 자동 저장 설정값이 잘못되면 기본값으로 fallback하며, 이 값만으로 실제 자동 저장 타이머나 DB 저장은 시작하지 않는다.

앱 시작 시 repository 계층을 통해 UI 설정을 읽고, 문자열 파싱에 실패하거나 값이 유효 범위를 벗어나면 기본값을 사용한다. 창 크기 기본값은 900x600이고 splitter 왼쪽 폭 기본값은 280px이며 UI 언어 기본값은 English다.

저장된 창 좌표가 현재 모니터 작업 영역과 겹치지 않으면 Windows 기본 위치로 보정해 창을 연다. Linux GTK4 백엔드는 저장된 창 크기와 splitter 위치를 복원하고, 창 위치는 Wayland 등 데스크톱 환경의 위치 제어 제한을 따른다. 저장된 선택 노드가 활성 트리에 없거나 삭제되었으면 첫 문서 또는 첫 노드로 fallback한다.

창 이동/크기 변경 완료, splitter 드래그 완료, 앱 종료, 활성 트리 선택 변경 시 현재 UI 상태를 저장한다. splitter 폭은 저장 시점의 현재 창 폭 안에서 편집 영역 최소 폭을 보존하도록 clamp한 값을 저장한다. 휴지통 보기에서 선택한 노드는 마지막 활성 트리 선택값을 덮어쓰지 않는다. 검색 결과가 없어서 선택 노드가 비어 있는 상태도 마지막 활성 트리 선택값을 지우지 않는다.

## Rich Edit 전환 후 릴리스 안정화 기준

Win32 본문 편집기가 Rich Edit 컨트롤로 전환된 뒤에도 릴리스 판정 기준은 문서 본문 저장 정책과 사용자 편집 흐름의 호환성 유지다. 본문 저장 형식은 계속 UTF-8 plain text이며, 검색창은 기존 Win32 Edit Control 동작을 유지해야 한다.

릴리스 전 검증 절차는 다음 체크리스트로 고정한다. 자동 검증이 실패하거나 수동 검증 중 본문 손상, 저장 형식 변경, 검색창 회귀가 발견되면 릴리스를 차단한다.

자동 검증:

- `cargo fmt --check`
- `cargo check`
- `cargo test`
- 가능한 경우 Linux GTK4 개발 패키지가 설치된 환경에서 `cargo check --target x86_64-unknown-linux-gnu`
- 가능한 경우 Windows Rust target과 링커가 설치된 환경에서 Windows target `cargo check`

Win32 수동 검증:

- 한글 입력: 본문 Rich Edit에 한글을 입력하고 저장 전후 글자 손실이나 조합 중 끊김이 없는지 확인한다.
- 이모지 입력: 본문 Rich Edit에 surrogate pair를 포함한 이모지를 입력하고 저장 후 재열기까지 보존되는지 확인한다.
- CRLF 여러 줄 문서: 여러 줄 본문이 CRLF 기준 plain text로 저장되고 다시 열 때 줄 수와 줄 경계가 유지되는지 확인한다.
- 찾기/바꾸기: 한글, 이모지, 여러 줄 본문에서 찾기와 바꾸기가 활성 편집 탭 기준으로 동작하고 선택 위치가 어긋나지 않는지 확인한다.
- 저장 후 재열기: 본문을 저장한 뒤 앱을 재시작하거나 탭을 다시 열어 plain text 본문만 복원되는지 확인한다.
- word wrap 토글: 자동 줄바꿈을 켜고 끄면서 본문 내용, dirty 상태, 가로 스크롤 동작이 정책과 일치하는지 확인한다.
- 글꼴 변경: 편집기 글꼴 변경이 본문 내용과 dirty 상태를 바꾸지 않고 오른쪽 본문 편집기에만 적용되는지 확인한다.
- 다크/커스텀 테마: 밝은 테마와 다크/커스텀 테마에서 본문 Rich Edit, 검색창, 트리, 탭의 배경과 글자색이 읽기 가능하게 적용되는지 확인한다.
- 읽기 전용 탭: 휴지통 문서 같은 읽기 전용 탭에서 본문 수정 명령은 차단되고 복사/전체 선택은 가능한지 확인한다.
- 검색창 동작: 검색창은 Rich Edit가 아니라 기존 Edit Control로 유지되며 입력, debounce 검색, 빈 검색어 복귀, 검색 결과 선택이 기존 정책과 일치하는지 확인한다.

Linux GTK4 수동 검증:

- 앱 시작: GTK4 UI가 같은 DB 위치 정책으로 시작하고 초기 문서가 표시되는지 확인한다.
- 메뉴/단축키: 파일, 편집, 문서, 보기, 도움말 메뉴와 단축키가 Win32 기준 기능 목록과 같은 결과를 만드는지 확인한다.
- 트리/검색/휴지통: 활성 문서, 검색 결과, 휴지통 전환, 트리 접기/펼치기와 선택 시 탭 열기가 같은 저장 확인 흐름을 따르는지 확인한다.
- 탭과 dirty 저장: 탭 열기, 닫기, 재정렬, dirty 탭 전환, 저장 충돌 선택지가 Win32와 같은 결과를 만드는지 확인한다.
- 파일 입출력: import/export/export all과 인코딩 선택이 같은 파일 내용을 만들거나 읽는지 확인한다.
- 글꼴/테마/언어/자동 줄바꿈: GTK4 위젯에서 설정이 저장, 복원되고 문서 본문이나 dirty 상태를 바꾸지 않는지 확인한다.

수동 검증 결과에는 실행 OS, 빌드 종류, 사용한 DB 경로, 실패 시 재현 단계와 실제 저장된 본문 샘플을 함께 기록한다.

## DB 위치 정책

DB 파일은 실행 파일과 같은 폴더에 만들고, 실행 파일의 확장자를 `.db`로 바꾼 이름을 사용한다. Linux처럼 실행 파일에 확장자가 없으면 파일명 뒤에 `.db`를 붙인다.

예: Windows `j3TreeText.exe` -> `j3TreeText.db`, Linux `j3TreeText` -> `j3TreeText.db`

DB 파일이 없으면 앱 시작 중 SQLite 연결 시 자동으로 생성하고, 마이그레이션을 적용한 뒤 초기 노드를 보장한다.

DB 파일을 열거나 만들 수 없으면 사용자에게 DB 경로를 보여주고 실행 파일 폴더의 쓰기 권한, 파일 잠금, 디스크 여유 공간을 확인하라는 메시지를 표시한다. 내부 SQLite 오류 원인은 사용자 메시지와 분리해 로그/상세 출력에 남긴다.

## CLI 정책

`j3TreeTextCli`는 같은 SQLite 문서 저장소를 조작하는 콘솔 프로그램이다. GUI와 동일한 `infra/sqlite` repository를 사용하며, 실행 시 마이그레이션과 최초 실행 초기 노드를 보장한다.

기본 DB 위치는 CLI 실행 파일의 확장자를 `.db`로 바꾼 경로다. 다른 DB를 조작해야 할 때는 전역 옵션 `--db <path>`로 명시한다.

CLI는 다음 작업을 제공한다.

- `tree`: 활성 문서 트리를 계층으로 출력한다.
- `show <node_id>`: 문서 제목, 부모, 수정 시각, 본문을 출력한다.
- `create --parent <node_id> --title <title>`: 지정 부모 아래 새 문서를 만든다. `--content`, `--content-file`, `--stdin` 중 하나로 초기 본문을 줄 수 있다.
- `edit <node_id>`: 문서 본문을 바꾼다. `--content`, `--content-file`, `--stdin` 중 하나를 사용하며, `--append`를 주면 기존 본문 뒤에 덧붙인다.
- `rename <node_id> --title <title>`: 문서 제목을 바꾼다.
- `search <query>`: 활성 문서의 제목과 본문을 검색한다.
- `delete <node_id>`: 문서와 하위 문서를 soft delete한다.
- `trash`: 휴지통 문서를 출력한다.
- `restore <node_id>`: 휴지통 문서와 삭제된 하위 문서를 복원한다.
- `purge <node_id>`: 휴지통 문서와 하위 문서를 영구 삭제한다.
- `move <node_id> --parent <node_id>`: 문서를 지정 부모의 마지막 자식으로 이동한다.
- `move-up <node_id>` / `move-down <node_id>`: 같은 부모 아래에서 문서 순서를 한 칸 이동한다.

CLI 편집은 실행 시점에 DB에서 읽은 최신 `updated_at`을 저장 충돌 토큰으로 사용해 조건부 저장한다. 저장 중 다른 프로세스가 먼저 같은 문서를 변경하면 GUI와 동일하게 저장 충돌 오류로 실패하며, 자동 덮어쓰기는 하지 않는다.
