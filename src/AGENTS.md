# Codex Agent Instructions (Rust 프로젝트)

이 문서는 Codex가 Rust 프로젝트에서 일관되게 작업하기 위한 기준입니다.
항상 한국어로 답변합니다. 코드/커밋 메시지/주석은 필요 시 영어를 섞어도 됩니다.

## 1) 핵심 원칙

1. 작은 프로젝트라도 유지보수 가능한 구조와 테스트 가능성을 우선합니다.
2. 엔트리포인트와 유스케이스와 외부 I/O의 의존 방향을 명확히 유지합니다.
3. 복구 가능한 오류는 `Result`로 전달하고, 내부 원인과 사용자 메시지를 분리합니다.
4. 런타임 경로에서 `unwrap()/expect()`와 근거 없는 `unsafe` 사용을 피합니다.
5. 동시성과 비동기 경계는 소유권, 취소, 종료 순서가 드러나게 설계합니다.
6. 새 크레이트는 신중히 추가합니다.
   - 표준 라이브러리 우선
   - 기존 크레이트 재사용 우선
   - 워크스페이스라면 가능한 한 `[workspace.dependencies]`로 관리
7. 소스 수정이 있으면 최소 검증까지 완료합니다.

## 2) 권장 구조

프로젝트 종류와 관계없이 역할이 섞이지 않도록 유지합니다.

- `entry`
  - `main.rs`, CLI, HTTP 핸들러, GUI 이벤트, 잡 실행기 등 진입점
- `app`
  - 상태 관리, 유스케이스 흐름, 트랜잭션/작업 경계 조합
- `domain`
  - 순수 규칙, 검증, 계산, 핵심 타입
- `infra`
  - 파일, 네트워크, DB, OS, 외부 서비스 연동
- `platform`
  - FFI, OS 전용 구현, 런타임/시스템 통합이 커질 때만 추가

소형 프로젝트 예시:

```text
src/
  main.rs
  app.rs
  domain.rs
  infra.rs
```

라이브러리 예시:

```text
src/
  lib.rs
  error.rs
  domain/
  infra/
```

워크스페이스 예시:

```text
crates/
  cli/
  app/
  domain/
  infra/
```

중요한 것은 폴더 수가 아니라, 진입점과 핵심 로직과 외부 I/O가 뒤섞이지 않도록 유지하는 것입니다.

## 3) 문서주도 DDD

- 기본 문서 파일은 `docs/domain.md`로 둡니다.
- 기능 변경 전 요구사항, 용어, 규칙을 먼저 문서에 짧게 정리합니다.
- 도메인 용어는 코드 이름과 문서에서 일관되게 사용합니다.
- 유스케이스 흐름은 `app`에 두고, 핵심 규칙은 가능하면 `domain` 또는 순수 함수로 분리합니다.
- 문서와 코드가 달라지면 코드 수정과 함께 문서를 갱신합니다.
- 작은 프로젝트에서는 과한 계층보다 명확한 책임 분리를 우선합니다.

## 4) Rust 규칙

- 공개 API와 모듈 경계에서는 소유권 이전, 참조, 수명 범위를 명확히 드러냅니다.
- 경계에서는 소유 타입을, 내부 핫패스에서는 필요한 범위의 참조를 우선 검토합니다.
- `clone()`으로 소유권 문제를 덮지 말고, 필요한 경우에만 비용을 감수합니다.
- 비동기 코드는 프로젝트가 이미 사용하는 런타임과 패턴을 우선 따릅니다.
- async 컨텍스트 안에서 블로킹 I/O, 긴 연산, 긴 락 점유를 직접 수행하지 않습니다.
- `Arc<Mutex<_>>`나 `RwLock`은 정말 공유 가변 상태가 필요한 경우에만 쓰고, 락 범위를 최소화합니다.
- 채널, 태스크, 스레드는 종료 시점에 안전하게 정리되도록 취소/종료 경로를 함께 설계합니다.
- `unsafe`는 꼭 필요한 범위로 고립하고, 불변식과 안전 근거를 코드 가까이에 남깁니다.
- feature flag는 가급적 additive하게 설계하고, 기본 동작을 불필요하게 깨지 않게 유지합니다.
- 애플리케이션/CLI/툴 프로젝트라면 `Cargo.lock`을 커밋에 포함합니다. 라이브러리는 저장소 정책을 우선합니다.

## 5) 에러 처리 / 로깅

- 복구 가능한 오류를 `panic!`으로 처리하지 않습니다.
- 라이브러리 성격의 코드에서는 구체적 에러 타입 또는 프로젝트 표준 에러 타입을 우선 사용합니다.
- 앱 경계에서는 프로젝트가 이미 쓰고 있다면 `anyhow` 같은 범용 에러를 사용할 수 있습니다.
- 에러 체인과 컨텍스트를 유지해 원인 추적이 가능하게 만듭니다.
- 내부에는 원인과 로그를 남기고, 외부에는 이해 가능한 메시지만 노출합니다.
- 로깅은 기본적으로 프로젝트 표준(`log`, `tracing` 등)을 따릅니다.

## 6) 플랫폼 및 경계 분리

- OS 전용 처리, FFI, 시스템 호출은 가능한 한 `infra` 또는 `platform`에 둡니다.
- 공용 로직과 `domain`에 `cfg` 분기를 남발하지 않습니다.
- 다른 target에서 컴파일 단계부터 깨지지 않도록 가드된 import, feature gate, 지연 초기화를 사용합니다.
- 공용 타입과 트레이트는 가능하면 `domain` 또는 별도 코어 모듈에 둡니다.
- 외부 라이브러리 타입이 상위 계층으로 새지 않도록 래핑 또는 변환 경계를 명확히 합니다.

## 7) 작업 방식

변경 전:
- 기존 크레이트 구조, 모듈 경계, feature, 에러 타입, 런타임 패턴을 먼저 확인합니다.
- 이미 있는 유틸, 모델, trait, 테스트 헬퍼를 우선 재사용합니다.

구현 시:
- 복잡한 로직은 함수나 모듈로 분리합니다.
- 매직 넘버와 설정값은 상수나 설정 구조체로 정리합니다.
- 공개 함수와 타입에는 필요 시 doc comment를 추가합니다.
- 한 번에 큰 변경보다 작은 단계로 나누어 수정합니다.
- 추측으로 구조를 바꾸지 말고, 확인된 문제와 목적에 근거해 수정합니다.

변경 후:
- 변경 파일
- 핵심 수정 내용
- 검증 결과
- 남은 리스크
를 함께 보고합니다.

## 8) 검증 기준

소스 수정이 있으면 아래 중 가능한 항목을 최소 1개 이상 수행합니다.

- `cargo check`
- `cargo build`
- `cargo test`
- `cargo fmt --check`
- `cargo clippy --all-targets --all-features -- -D warnings`

프로젝트가 특정 feature, target, workspace, CI 규칙을 이미 쓰고 있으면 그 규칙을 우선합니다.
가능하면 관련 바이너리나 라이브러리 사용 흐름의 스모크도 확인합니다.

## 9) 금지 사항

- 런타임 경로에서 `unwrap()/expect()` 남용
- 근거 없는 `unsafe` 사용
- async 런타임에서 블로킹 작업 수행
- 필요성 없는 `clone()`과 공유 가변 상태 남용
- 기존 패턴을 무시한 중복 구현
- 의존성 버전과 선언 위치를 무질서하게 분산
- 플랫폼 전용 코드를 공용 로직에 무분별하게 혼합
- `.my` 폴더 접근 또는 분석

추가 제외 경로/패턴:

`target/`, `.git/`, `.my/`, `.idea/`, `.vscode/`, `dist/`, `coverage/`, `criterion/`, `tarpaulin-report.html`, `cargo-tarpaulin-report.xml`, `flamegraph.svg`, `*.rlib`, `*.rmeta`, `*.profraw`, `*.profdata`, `*.pdb`, `*.ilk`, `*.log`, `*.tmp`, `*.bak`, `.DS_Store`, `Thumbs.db`, `Desktop.ini`, `*~`, `*.swp`, `*.swo`

위 경로/패턴과 그 하위 항목은 기본적으로 모든 탐색·인용·계획·수정 대상에서 제외한다.
단, 사용자가 해당 제외 경로 내부 작업을 명시적으로 요청한 경우에만 그 범위에서 허용한다.

## 과도한 설계 피하기

깔끔하고 읽기 쉽고 유지보수하기 쉬운 코드를 작성하되, 불필요한 설계는 피하십시오.

중복, 실제 변경 압박, 테스트 용이성 또는 기존 프로젝트 패턴으로 정당화되기 전까지는 단순하고 구체적인 구현을 우선하십시오.

미래에 필요할 수도 있다는 이유만으로 인터페이스, 팩토리, 전략 패턴, 제네릭 레이어, 래퍼, 설정 기반 구조를 도입하지 마십시오.

모든 추상화는 현재 코드베이스 안에서 구체적인 이유가 있어야 하며, 추가할 경우 최소한으로 유지하고 주변 코드와 일관되게 작성하십시오.
